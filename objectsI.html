Each flower is an OBJECT

We already know that each of our flowers in the flowers array is a JavaScript object, created using createFlower().
That object keeps track of the specific properties of a flower (like its position and colors).

Objects that are created by writing out the properties inside curly brackets as in createFlower() are technically called JavaScript object literals.
(Because we literally write out all the properties AT THE SAME TIME AS WHEN we create them.)


However, in our program there are other places that are directly related to our flowers:

  • The createFlower() function is devoted to creating a flower
  • The renderFlower() function is devoted to rendering a flower to the browser window

So those parts of our code basically belong to the flower objects.
They're essentially things that a flower can DO (be created and be displayed).

Combining properties and functions

The fact that in our program we have both properties that provide information about flowers and functions for dealing with flowers:
leads toward another way of thinking about these flowers (and JavaScript Objects in general).


In particular, it might be nice to be able to combine the properties and functions related to a flower into one place .

This would potentially make our program even clearer, more modular, and perhaps more reusable.


This is the idea behind Object-Oriented Programming (OOP).

In OOP we think about our program as being explicitly organized according to different kinds of objects,
each of which has its own set of properties (which we already do) but also its own set of functions that control how it behaves.

To do this we write classes that describe how a particular kind of object (like a flower) behaves (its properties and functions).

A class is like a template/ blueprint for an object.

Creating a Flower class

So, let's write a class called Flower.

We're going to create this class inside a new file which we will name after the class.

The file will be called Flower.js.

Any time we want to create a class, we will make a new file in our /jsfolder with the same name as the class. So, we create Flower.js.

To begin with, we'll include the most basic skeleton for defining a class, which looks like this:


class Flower {
    
}
  • class - this tells JavaScript we are going to define a class
  • Flower - this is the name of our class (by convention this begins with a capital letter, unlike a variable name or a function name)
  • { ... } - we use curly brackets to define a block of code that will contain our definition of the Flower class

Linking the class

In order to actually use our class later on, we're going to need our main program in garden.js to know that class is there.

It doesn't by default.

To do this, we have to go into our index.html to add our Flower.js file to the set of scripts (JavaScript) that our project knows about.

Importantly, our new script should be added before the garden script.

As a rule of thumb, you should add new scripts so that they come before any script that needs to use them.

We put Flower.js before garden.js, because the main program in garden.js will be using the class we're going to define in Flower.js
(it needs to create and display flowers which will rely on the Flower class in Flower.js).


<script src="js/Flower.js"></script>
<script src="js/garden.js"></script>

This class will be used to create new Flower objects that will be part of our running program.

Our class describes how these objects will work when they are created.

In order to create a new Flower object from our class, the first thing we will need is a special function that will construct (create) the Flower when asked.
It's called the constructor:


class Flower {
    constructor() {
        // We write instructions to set up a Flower here
      }
}
  • We do not write function in front of the constructor function, even though we are clearly defining a function.
    Because it's inside our class definition for Flower, it's already understood by JavaScript that we are defining functions.
  • Terminology: we don't actually call the functions in a class functions, we call them methods.

But how????

In the constructor, we want to write code that can set up a prospective Flower with the properties it needs to work -> === the createFlower() function!.
Our constructor will be highly similar to createFlower(), with a couple of key differences:

  1. No need to declare a variable to store the object, there is a special variable called this that automatically refers to the flower being created in the constructor.
  2. No need to return the flower being created because the constructor just does that automatically.

Here's what it looks like (notice the similarity to createFlower()):


class Flower {
  constructor() {
    // We write instructions to set up a Flower here
    // Position and size information
    this.x = Math.random() * (window.innerWidth);
    this.y = Math.random() * 120;
    this.size = 40;
    this.stemLength = 75;
    this.stemThickness = 10;
    this.petalThickness = 8;
    this.flowerStemDiv = document.createElement("div");
    this.flowerPetalDiv = document.createElement("div");

    // Color information
    this.stemColor = {
      r: 50,
      g: 150,
      b: 50,
    };
    this.petalColor = {
      r: 200,
      g: 50,
      b: 50,
    };
    this.centreColor = {
      r: 50,
      g: 0,
      b: 0,
    };
  }
}

The biggest difference you probably notice here is the use of the word this.
In a class definition, this refers to the object that will be created with the class.
So in the constructor, this means this object we are constructing from the class.

we need to add a method,(remember this is just the name for a function inside a class) to the Flower class.

This code will be very similar to the code we used in renderFlower() in our initial program!

The renderFlower() method here is identical to the renderFlower() function we used previously, except:
  • Don't write the keyword function before renderFlower() (because we're in a class and JavaScript understands we're defining a method)
  • No need to have a flower parameter-> use this to refer to the flower being rendered
  • Use this to access the properties of the Flower set up in the constructor()

Finally - create a flower!

In Summary:

  • A complete Flower class is defined in Flower.js.
  • It replicates all the properties and functions as in the initial program.
  • It sets all the the same properties from createFlower() in its constructor()
  • It does all the same things in renderFlower() that we were doing in renderFlower() initially...

To actually use this class, we need to create (construct) flowers with it.

To do this we use another special word called new. We do this back in our main script (garden.js):


  // Create our flowers by counting up to the number of the flowers
    for (let i = 0; i < garden.numFlowers; i++) {
      // NEW! Create a new flower
      let flower = new Flower();
      // Add the flower to the array of flowers
      garden.flowers.push(flower);
    }
The big difference here is that we have replaced: 

let flower = createFlower();
with 

let flower = new Flower();

Let's look at the pieces of the new version:

  1. let tells JavaScript a new variable is being created
  2. flower is the name of the variable being creating to temporarily store a new flower in before being put into the array
  3. = is used to assign the new flower into the variable
  4. new tells JavaScript a new object is going to be created using a class definition
  5. Flower() tells JavaScript to use the constructor() of the Flower class to create a new object

Note: the constructor() is not explicitly called. Why? -> because when the name of the class is used along with the new keyword - the constructor is being called automatically.

I see no flowers in the browser ?!

You have all your flowers objects, created with the Flower class, in the array.

You just need to display them.

The Flower class has a renderFlower() method, so let us use that to display all our flowers in a for loop that goes through the array.

To call a method that is part of an object created with a class, use dot notation just like when accessing an object's properties:


for (let i = 0; i < garden.numFlowers; i++) {
  
    // render the array of flowers
    garden.flowers[i].renderFlower();
  }

We did it!

The garden is back! This time it's being done through a class called Flower! How?

  • A class called Flower was defined which provides a constructor() to set all the properties of a flower
    and a renderFlower() method to render a flower in the browser.
  • New Flower objects were created using the new keyword and the constructor (written as Flower()), storing them in the array.
  • The renderFlower() method of the flowers was called to render them to the browser.

But why do it this way?

Using OOP in this way by creating a class brings us back to the concepts of modularity and reusability.

By consolidating all the aspects of a flower into one location in a separate file, the Flower class is really helpful.

If you ever want to change what a Flower does or what properties it has, you can just change the class.

This is a highly modularway of writing this program. And, the Flower class is reusable.


We made many Flower objects and stored them in our array, with each one working in the way the Flower class defines.

Note that the result is still functionally the same as the initial program.

It's ultimately just a different way of expressing the same ideas.

But it's often a very satisfying and clear way to write a program because it so explicitly separates different objects out into classes.
