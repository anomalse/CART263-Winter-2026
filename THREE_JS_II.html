INTRO TO THREE JS II

CUSTOM CONTROLS

This part of the lecture will consider how to get the camera be controlled by some input device (i.e. the mouse).
You could build the functionality from scratch (using event lsiteners etc) ... or you can use the inbuilt control objects provided by Threee.js.

  • FlyControls enable moving the camera like if you were on a spaceship. You can rotate on all 3 axes, go forward and go backward.
  • FirstPersonControls is just like FlyControls, but with a fixed up axis. For example, you could see like a flying bird view where the bird cannot do a barrel roll.
    While the FirstPersonControls contains the word FirstPerson, it doesn't work like in FPS (First Person shooter) games.
  • PointerLockControls uses the Pointer Lock JavaScript API.
    This API hides the cursor, keeps it centered, and keeps sending the movements in the mousemove event callback.
    With this API, you can create FPS games right inside the browser. While this class sounds very promising if you want to create that kind of interaction, it'll only handle the camera rotation when the pointer is locked. You'll have to handle the camera position and game physics by yourself.
  • OrbitControls performs orbiting, zooming, and panning. You can rotate around a point with the left mouse, translate laterally using the right mouse, and zoom in or out using the wheel.
  • TrackballControls is just like OrbitControls but there are no limits in terms of vertical angle. You can keep rotating and do spins with the camera even if the scene gets upside down.
  • TransormControls has nothing to do with the camera. You can use it to add a gizmo to an object to move that object.
  • DragControls has nothing to do with the camera either. You can use it to move objects on a plane facing the camera by drag and dropping them.

RAYCASTiNG

So ... onto a more advanced but necessary topic: RayCasters and RayCasting.
As the name suggests, a Raycaster can cast (or shoot) a ray in a specific direction and test what objects intersect with it.
You can use that technique to detect if there is a wall in front of the player, test if a weapon hit something, test if something is currently under the mouse to simulate mouse events, and many other things.

RAYCASTiNG & MOUSE EVENTS

We can also use the raycaster to test if an object is behind the mouse.
In other words, if you are hovering an object :). Mathematically speaking, it's a little complex because we need to cast a ray from the camera in the mouse's direction, but, fortunately, Three.js does alot of the work for us.

Mouse Hover

To begin with, we need the coordinates of the mouse. We cannot use the basic native JavaScript coordinates, which are in pixels -
We need a value that goes from -1 to +1 in both the horizontal and the vertical axis, with the vertical coordinate being positive when moving the mouse upward.
This is how WebGL works and it's related to things like clip space but we don't need to understand those complex concepts ;)

In essence we need to map the following:

  • The mouse is at the top left of the canvas: Horiz: -1 Vertical : 1
  • The mouse is at the bottom left of the canvas: Horiz: -1 Vertical : -1
  • The mouse is at the middle vertically and right horizontally of the canvas: Horiz: 1 Vertical : 0
  • The mouse is at the middle vertically and left horizontally of the canvas: Horiz: -1 Vertical : 0
  • The mouse is at the center of the canvas: Horiz: 1 Vertical : 0
  • ...

  • Lets go! - use the same code as before - except lets clean out the animate() to be: 
      
    function animate(timer) {
    controls.update();
    renderer.render(scene, camera);
    window.requestAnimationFrame(animate);
 }
  • Next, we will create a variable to hold the mouse coordinates using a Vector2, and an event listener callback to capture the mousemove: 
      
  window.addEventListener("mousemove", function(event) {
  mouse.x = (event.clientX / sizes.width) * 2 - 1; //map to between -1,1
  mouse.y = -(event.clientY / sizes.height) * 2 + 1; //map to between -1,1
  //console.log(mouse);
  });
  • We could cast the ray in the mousemove event callback, but it's not recommended because the mousemove event might be triggered
    more than the frame rate for some browsers... Instead, we will cast the ray in the animate() function as we did before.
    To orient the ray in the correct direction, we can use the Three.js setFromCamera() method on the Raycaster.
    The rest of the code will be the same as before - and we will also update the objects materials to red or blue if they intersect or not:

    object.material.color.set("#ff0000");

Ambient light

The Ambient Light applies omnidirectional lighting on all geometries of the scene. The first parameter is the color and the second parameter is the intensity.


const ambientLight = new THREE.AmbientLight()
scene.add(ambientLight)

You can set the color and intensity like this: 


ambientLight.color = new THREE.Color(0xff0000)
ambientLight.intensity = .5;

Directional light

The Directional lightwill have a sun-like effect as if the sun rays were traveling in parallel.
The first parameter is the color and the second parameter is the intensity.
By default, the light will seem to come from above. To change that, you need to move the whole light by using the position property as with any other Three.js object:


 directionalLight.position.set(-5,5, 0)

we can of course move the directional light - in our animate():


 let x = directionalLight.position.x
  x+=.02
  directionalLight.position.set(x,5, 0)

Point Light

The Point Light is like a lighter. The light source is infinitely small, and the light spreads uniformly in every direction.
The first parameter is the color and the second parameter is the intensity:

  
  const pointLight = new THREE.PointLight(0xff9000, 1.5)
  scene.add(pointLight)
  console.log(pointLight.position) // default position is 0,0,0

Of course you can change the position of the light :)

  
    pointLight.position.set(0,1,0) //set y 

    //or
     //pointLight.position.set(1,1,1) 

     //set the intensity too
      pointLight.intensity = 5

The point light has a third parameter for distance. Its default value is 0, meaning that the distance is actually infinite.
Try the following:

  
     pointLight.distance = 1

The light looks quite similar, but the intensity drops down drastically when getting close to the 1.0 distance cutoff point/threshold.
In case you wonder what this distance number means: here is a link to the inverse-square law.

Finally, the point light also has a decay parameter.The default value is 2.
Try setting it to 0.5. Also, set the distance back to 0.The lower the decay, the faster the light will decay.

  
    pointLight.distance =0
    pointLight.decay = .5

Spot Light

The Spot light works like a flashlight. It's a cone of light starting at a point and oriented in a direction. Here the list of its parameters:

  • color: the color
  • intensity: the strength
  • distance: the distance at which the intensity drops to 0
  • angle: how large is the beam
  • penumbra: how diffused is the contour of the beam
  • decay: how fast the light dims

Let's create one:


const spotLight = new THREE.SpotLight(0x78ff00, 4.5, 10, Math.PI * 0.1, 0.25, 1)
spotLight.position.set(0, 2, 3)
scene.add(spotLight)

The Spot light has a property called target - which is of type Object3D. The Spotlight is ALWAYS looking at that target object which by default is (0,0,0).
In order to successfully change the target's position/rotation it has to be added manually to the scene (eventhough its an empty object)


scene.add(spotLight.target)

And now we can set a new target: i.e. 


spotLight.target.position.x = -2

Positioning and orienting the lights is hard. Three,js provides the following helper objects:

  • HemisphereLightHelper
  • DirectionalLightHelper
  • PointLightHelper
  • SpotLightHelper

Casting Shadows

Now that you have lights - you may possibly also want shadows ;) The back of the objects are in the dark, and this is called the core shadow. What we are missing is the drop shadow, where objects create shadows on the other objects.
Note: - Creating realistic lighting and shadows is hard and the techniques that will be shown next are built in solutions from Three.js - they are convenient but far from perfect.

The basics of how shadows work in Three.js:

  • When you do one render, Three.js will first do a render for each light that is supposed to cast shadows.
  • Those renders will simulate what the light 'sees' as if it were a camera.
  • During these lights renders, the MeshDepthMaterial replaces all materials for those meshes.
  • The results are then stored as textures and named shadow maps.
  • You won't see those shadow maps directly, but they are used on every material supposed to receive shadows and projected on the geometry.
  • Here's an excellent example of what the directional light and the spotlight see: this link

Let's start with a new starter js: shadows.js:


import * as THREE from "three";
import { OrbitControls } from "three/addons/controls/OrbitControls.js";

const scene = new THREE.Scene();
// Canvas
const canvas = document.querySelector("canvas#three-ex");

//Ambient Light
const ambientLight = new THREE.AmbientLight(0xffffff, 1);
scene.add(ambientLight);

// Directional light
const directionalLight = new THREE.DirectionalLight(0xffffff, 1.5);
directionalLight.position.set(2, 2, -1);
scene.add(directionalLight)

//Sphere and plane
const geometry = new THREE.SphereGeometry(0.5, 32, 32);
const material = new THREE.MeshStandardMaterial({});
material.roughness = 0.7;

const sphere = new THREE.Mesh(geometry, material);
scene.add(sphere);

const plane = new THREE.Mesh(new THREE.PlaneGeometry(5, 5), material);
scene.add(plane);
plane.rotation.x = -Math.PI * 0.5;
plane.position.y = -0.5;

const sizes = {
  width: window.innerWidth,
  height: window.innerHeight,
};
const camera = new THREE.PerspectiveCamera(
  75,
  sizes.width / sizes.height,
  0.1,
  100,
);
camera.position.x = 1;
camera.position.y = 1;
camera.position.z = 2;

scene.add(camera);

// Controls
const controls = new OrbitControls(camera, canvas);
controls.enableDamping = true;

//RENDER
const renderer = new THREE.WebGLRenderer({
  canvas: canvas,
});
renderer.setSize(sizes.width, sizes.height);


//ANIMATE
window.requestAnimationFrame(animate);
function animate() {
  controls.update();

  renderer.render(scene, camera);
  window.requestAnimationFrame(animate);
}
  • Start by activating the shadow maps on the renderer: 
      
    renderer.shadowMap.enabled = true
  • Then, we need to go through each object in the scene and decide if the object can cast a shadow with the castShadow property, and if the object can receive shadows with the receiveShadow property. Try to activate these on the least amount of objects as possible (computationally expensive) 
      
    sphere.castShadow = true
    plane.receiveShadow = true
  • Finally - we want to activate the shadows on the light with the castShadow property - again try to activate on as few lights as possible. Only the following lights support shadows:
    • Point Light
    • Directional Light
    • Spot Light
    Let's add to the directional light: 
        
    directionalLight.castShadow = true

Shadow map optimisations

Ok - we want to perhaps attempt to improve our shadow somewhat ... First of all - as the directional Light is the one casting the shadows - we can log the properties: 

console.log(directionalLight.shadow)

By default, the shadow map size is only 512x512 for performance reasons. We can improve it but it needs to be values that are powers-of-2:


directionalLight.shadow.mapSize.width = 1024
directionalLight.shadow.mapSize.height = 1024

Three.js uses cameras to implemnet the shadow map renders. Those cameras have the same properties as the cameras you have implemented:)
So - we can also modify/configure/set the camera near and far values. It won't really improve the shadow's quality, but it might fix bugs where you can't see the shadow or where the shadow appears suddenly cropped.

We will not go into the implementation here (lack of time) - but know that it is always a possibility :)

You can control the shadow blur with the radius property:


directionalLight.shadow.radius = 10

This technique doesn't use the proximity of the camera to the object. It's just a general and cheap blur.

Shadow map algorithms

Different types of algorithms can be applied to shadow maps:

  • THREE.BasicShadowMap: Very performant but not-good quality
  • THREE.PCFShadowMap Less performant but smoother edges
  • THREE.PCFSoftShadowMap: Less performant but even softer edges
  • THREE.VSMShadowMap: Less performant, more constraints, can have unexpected results

To change it, we just need to update the renderer.shadowMap.type property. The default is THREE.PCFShadowMap.
To use THREE.PCFSoftShadowMap for better quality:


renderer.shadowMap.type = THREE.PCFSoftShadowMap

Three.js Shadows can be very super-fragalistic if the scene is simple - but otherwise can become quite cumbersome...
A good alternative is baked shadows - where the Shadows are integrated into textures and then applied on the materials of the relevent objects.

The same strategy can be used for the lights themselves: you bake the light into the texture of the object. This can be done in a 3D software like Blender.
The downside is that, once the lighting is baked, you can't move the lights, because there are none and you'll probably need a lot of textures....

BLENDER MODELS

Three.js lets you create many primitive geometries, but when it comes to more complex shapes, it is easier to use a dedicated 3D software - i.e. Blender.
The main question then is what format do we use to export from Blender? There are hundreds of model formats.

GLTF

GLTF stands for GL Transmission Format. It's made by the Khronos Group. GLTF is a format that has become very popular these past few years.
It supports very different sets of data. You can have data like the geometries and the materials but you can also have data like cameras, lights, scene graphs, animations, skeletons... GLTF has become the standard - and so most 3D softwares, game engines, and libraries support it. That means that you can acheive similar results in different environments.

Get The sample model

In this lesson, we will use sample assets sourced from here
We will start with the fox :) You can also just download the folder here.
While GLTF is a format in itself, it can have different file formats too. It's a little complex but for good reasons. If you open the Fox folder, you'll see 3 different folders. Each one contains the fox but in different GLTF formats.

  • glTF
  • glTF-Binary
  • glTF-Embedded

glTF

This format is basically the default format. The Fox.gltf file is a JSON that you can open in your editor. It contains various information like cameras, lights, scenes, materials, objects transformations, but neither the geometries nor the textures.
The Fox.bin file is a binary that you can't read like this. It usually contains data like the geometries and all information associated with the vertices like UV coordinates, normals, vertex colors, etc.
The Texture.pngis simply the texture of the fox.
When we load this format, we only load the Fox.gltf that contains references to the other files that will then be loaded automatically.

glTF-Binary

This format is composed of only one file. It contains all the data as in the default format. It is a binary file, so you can't just open it in your code editor to see what's inside. This format can be a little lighter and more comfortable to load because there is only one file, but you won't be able to easily alter its data.
For example, if you want to resize or compress the textures, you can't because it's inside the binary file.

glTF-Embedded

This format is like the glTF-Binary format because it's only one file, but this file is actually a JSON that you can open in your editor. The only benefit of this format is to have only one easily editable file.
